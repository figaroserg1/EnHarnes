Harness Engineering — как построить agent-first разработку (по мотивам OpenAI)

OpenAI описали очень сильный подход: команда строит продукт, где люди не пишут код руками, а проектируют среду, правила, инструменты и обратную связь, чтобы агент (Codex) делал работу сам. По их оценке, это дало примерно ~10x ускорение в сравнении с ручной разработкой в этом эксперименте. За ~5 месяцев — около 1M строк, ~1500 PR, и всё это маленькой командой. 

Ключевая идея:
Humans steer. Agents execute.
(Люди управляют и задают рамки; агенты исполняют.) 


---

Что важно понять сразу (чтобы не повторять типичную ошибку)

Это не “дайте ИИ промпт и он сам всё сделает”.
Это подход, где вы строите инженерный harness (обвязку):

понятная структура репозитория

короткий AGENTS.md как карта, а не энциклопедия

docs внутри репо как single source of truth

механические проверки (линтеры, структурные тесты, CI)

инструменты наблюдаемости, доступные агенту

способность агента запускать приложение, тестировать UI, читать логи/метрики

цикл авто-ревью и исправлений PR агентом до “зелёного” состояния 



---

Быстрый чеклист “как сделать так же”

1) База

Git-репозиторий (с нуля — ок)

Codex CLI / coding agent CLI

gh (GitHub CLI)

CI (GitHub Actions / аналог)

единый package manager / build tool

форматирование + линт + тесты (обязательно)

git worktree (для параллельных задач)


2) Агентная документация

короткий AGENTS.md (примерно ~100 строк, как оглавление)

ARCHITECTURE.md (карта кода / слоёв)

docs/ как системный источник знаний

exec-plans / PLANS для сложных задач

индексированные design docs / product specs / tech debt / quality score


3) “Видимость” для агента (legibility)

агент должен уметь:

запускать приложение

открывать UI

снимать DOM snapshot / screenshot

ходить по экранам (через CDP / DevTools)

читать логи, метрики, трейсы


локальная observability-обвязка на каждую worktree-задачу (эпhemeral)


4) Жёсткие рамки

архитектурные слои + запреты зависимостей

custom linters + structural tests

“taste invariants” (логирование, naming, лимиты на размер файлов и т.п.)

CI, который проверяет не только код, но и docs / cross-links / свежесть знаний


5) Процесс

человек задаёт цель → агент делает PR

агент сам себя ревьюит + запускает дополнительные agent review

цикл замечаний → фиксов → проверок

merge philosophy: короткоживущие PR, быстрые корректировки follow-up’ами


6) Обслуживание энтропии

“doc gardening” агент (чистка устаревших доков)

регулярные cleanup-задачи

golden principles в коде и линтерах

непрерывное снижение техдолга маленькими порциями



---

Почему это работает

OpenAI прямо пишут, что узким местом стала не генерация кода, а человеческое QA-время. Поэтому они начали делать приложение, UI и observability читаемыми для агента: агент сам воспроизводит баги, валидирует фиксы, анализирует метрики, открывает PR. 

Главный сдвиг роли инженера:

раньше: писать код

теперь: проектировать среду, ограничения, инструменты, feedback loops 



---

Практический мануал: как воспроизвести подход в своём проекте

Шаг 0. Принцип (обязательный)

Сразу зафиксируйте для команды:

> Наша цель — не “сгенерировать код”, а “построить систему, в которой агент стабильно делает полезную инженерную работу”.



Если не зафиксировать это — всё скатится в хаотичные промпты и ручные доработки.


---

Шаг 1. Стартовый стек (инструменты)

Минимум, который должен быть у агента под рукой (локально или в CI):

git

gh (GitHub CLI) — OpenAI прямо упоминают, что агент использует стандартные dev tools, включая gh 

ваш runtime/toolchain (Node/Python/Go/…)

тест-раннер

линтеры / форматтер

скрипты запуска приложения

скрипты сидирования данных / фикстур

скрипты сборки и smoke-check

скрипты для локального observability стенда


Идея: агент не должен зависеть от копипаста человека в чат. Он должен сам брать контекст и сам запускать команды. 


---

Шаг 2. Структура репозитория (главное)

OpenAI показали очень сильную схему: AGENTS.md — это карта, а не свалка правил; все знания лежат в docs/ и версионируются в репо. 

Ниже — практический шаблон (можно брать почти как есть):

/
├─ AGENTS.md                 # Краткая карта для агента: как работать, куда смотреть
├─ ARCHITECTURE.md           # Высокоуровневая карта доменов/слоёв/границ
├─ README.md                 # Для людей (не дублировать агентные детали)
├─ .github/
│  └─ workflows/             # CI (code + docs + structure checks)
├─ scripts/
│  ├─ dev-start.sh|ps1       # Запуск приложения
│  ├─ test-all.sh|ps1        # Все проверки
│  ├─ lint-all.sh|ps1
│  ├─ seed-dev-data.sh|ps1
│  ├─ obs-up.sh|ps1          # Поднять локальный observability stack
│  └─ obs-down.sh|ps1        # Снести после задачи
├─ tools/
│  ├─ agent-skills/          # Репозиторные skills / helper scripts для агента
│  ├─ linters/               # Custom linters (архитектурные правила, naming, и т.д.)
│  └─ structural-tests/      # Тесты структуры зависимостей и слоёв
├─ docs/
│  ├─ design-docs/
│  │  ├─ index.md
│  │  ├─ core-beliefs.md
│  │  └─ ...
│  ├─ exec-plans/
│  │  ├─ active/
│  │  ├─ completed/
│  │  └─ tech-debt-tracker.md
│  ├─ generated/
│  │  └─ db-schema.md        # Автогенерируемые карты схем / API и т.п.
│  ├─ product-specs/
│  │  ├─ index.md
│  │  ├─ onboarding.md
│  │  └─ ...
│  ├─ references/
│  │  ├─ framework-llms.txt
│  │  ├─ infra-llms.txt
│  │  └─ ...
│  ├─ DESIGN.md
│  ├─ FRONTEND.md
│  ├─ PLANS.md
│  ├─ PRODUCT_SENSE.md
│  ├─ QUALITY_SCORE.md
│  ├─ RELIABILITY.md
│  └─ SECURITY.md
├─ app/ or src/
└─ tests/

Почему так:

агенту нужен прогрессивный доступ к контексту, а не “1000-страничная инструкция” в одном файле 

знания вне репо (Google Docs, чаты, “в голове”) для агента не существуют 



---

Шаг 3. Что писать в AGENTS.md (и чего НЕ писать)

OpenAI пришли к выводу, что “один большой AGENTS.md” плохо работает: он переполняет контекст, быстро устаревает и его трудно валидировать. Поэтому AGENTS.md должен быть оглавлением. 

Что положить в AGENTS.md:

как поднять проект

как прогнать проверки

где архитектурная карта (ARCHITECTURE.md)

где product specs / design docs / reliability / security

как оформлять PR

что делать перед merge

куда писать execution plans для больших задач


Что не делать:

не превращать AGENTS.md в энциклопедию

не копировать туда всё из docs

не хранить “устные договорённости” только в чатах


Отдельно: сам формат AGENTS.md — открытый и поддерживается многими инструментами/агентами. Это не закрытая штука. 


---

Шаг 4. ARCHITECTURE.md — карта, а не роман

В статье OpenAI они ссылаются на идею ARCHITECTURE.md как на верхнеуровневую карту доменов и package layering. 

Практически:

кратко описать домены

перечислить слои

указать допустимые направления зависимостей

явно назвать архитектурные инварианты

описать cross-cutting concerns (auth, telemetry, feature flags, connectors)


Важно: цель ARCHITECTURE.md — отвечать на вопрос
“где находится X и через какие границы туда идти?”
а не дублировать реализацию.


---

Шаг 5. PLANS / execution plans для сложных задач

OpenAI выделяют планы как первоклассный артефакт: маленькие изменения — лёгкие планы, большие задачи — полноценные execution plans с логом решений и прогресса прямо в репо. 

Практика:

docs/exec-plans/active/…

docs/exec-plans/completed/…

в плане держать:

цель

acceptance criteria

шаги

риски

решения/компромиссы

журнал прогресса

что проверять агенту перед PR



Это резко снижает зависимость от “внешнего контекста”.


---

Шаг 6. Сделайте приложение “читаемым” для агента (самое недооценённое)

В статье это один из ключевых инсайтов.

OpenAI:

сделали приложение запускаемым на каждый git worktree

подключили Chrome DevTools Protocol

добавили skills для DOM snapshots, скриншотов и навигации 


Зачем:

агент может сам воспроизводить баг

сам валидировать фикс в UI

сравнивать состояния до/после

запускать цикл “исправил → перезапустил → перепроверил”


Что повторить у себя

Поддержка git worktree для параллельных задач

Команда start-dev должна запускать isolated instance

UI automation / CDP-мост для агента

Скрипты “сделать screenshot / собрать DOM snapshot / пройти сценарий”



---

Шаг 7. Observability, доступная агенту

OpenAI дали агенту локальный ephemeral observability stack для каждого worktree и возможность запрашивать:

логи (LogQL)

метрики (PromQL)

трейсы (TraceQL)
через локальный стек (в статье упомянуты Vector и Victoria Logs/Metrics/Traces). 


Почему это критично:

можно ставить задачи типа “startup < 800ms” или “ни один span в 4 ключевых user journeys > 2s” — и агент реально может это проверить. 


Минимальный practical setup

локальный collector / forwarder

локальные endpoints логов/метрик/трейсов

скрипты поднятия/сноса стенда на задачу

репозиторные команды, которые агент может вызвать сам

smoke-скрипты, генерирующие нагрузку/сценарии



---

Шаг 8. Архитектурные ограничения и “taste invariants”

OpenAI подчёркивают: документации недостаточно. Нужны механически проверяемые инварианты. 

Они описывают:

фиксированные слои домена

строгие направления зависимостей

ограниченный набор допустимых связей

custom linters + structural tests

“taste invariants” (structured logging, naming, file size limits, reliability rules и др.) 


Что это значит на практике

Не пишите “пожалуйста, соблюдай архитектуру” в промпте.
Сделайте так, чтобы нарушение:

ловилось линтером

ломало CI

давало агенту понятное сообщение с инструкцией, как исправить


OpenAI отдельно отмечают, что текст ошибок линтера можно писать так, чтобы он сразу становился remediation-подсказкой для агента. 

Это супер-практика.


---

Шаг 9. Процесс работы: “Ralph Wiggum loop” / цикл агент-ревью

OpenAI описывают процесс так:

инженер формулирует задачу

агент делает изменения и открывает PR

агент локально ревьюит себя

запрашивает доп. agent reviews (локально/в облаке)

отвечает на фидбек

повторяет цикл, пока ревьюеры не удовлетворены 


Идея для воспроизведения:

шаблон промпта на PR-цикл

список обязательных проверок

команда “самопроверки”

команда “ответить на комментарии”

команда “перезапустить failing checks и исправить”



---

Шаг 10. Merge philosophy меняется

OpenAI пишут, что при высокой агентной пропускной способности часть классических норм начинает мешать:

минимальные блокирующие merge-gates

короткие PR

flaky tests часто чинятся follow-up’ами, а не бесконечной блокировкой процесса 


Важный нюанс: это не универсальный совет. Они прямо говорят, что такой trade-off был бы безответственным в low-throughput среде. 


---

Шаг 11. Энтропия и “garbage collection” кода

Очень сильный фрагмент статьи: с полной агентной автономией возникает новая проблема — агент копирует существующие паттерны, включая плохие. Это ведёт к дрейфу. 

OpenAI сначала чистили это вручную (по пятницам, ~20% времени), потом перевели в систему:

“golden principles” в репо

регулярные cleanup-задачи агента

обновление quality grades

точечные refactoring PR, часто авто-мержимые 


Вывод: техдолг в agent-first мире нужно обслуживать непрерывно, а не “генеральной уборкой раз в квартал”.


---

Шаг 12. Что именно должен делать человек (новая роль инженера)

OpenAI очень чётко переопределяют роль инженера. Люди остаются в контуре, но на другом уровне:

приоритизация

превращение фидбека пользователей в acceptance criteria

валидация результата

проектирование инструментов/ограничений/документации

эскалации, где нужен judgment 


Если агент “тупит” — это не повод просто переписать код руками.
Это сигнал: какая capability отсутствует?

инструмент?

guardrail?

документация?

проверка? И это нужно добавить в систему. 



---

Минимальный шаблон запуска (MVP harness) — чтобы начать за 1–2 недели

Если хотите повторить подход без overkill, начните так:

Неделя 1

создать структуру AGENTS.md + ARCHITECTURE.md + docs/

завести CI на:

test

lint

format

docs link/style checks


добавить custom structural test (хотя бы 1 правило)

перевести архитектурные договорённости из чатов в repo docs


Неделя 2

дать агенту команды запуска проекта + тестов + линтов

добавить git worktree workflow

подключить UI automation (или хотя бы e2e smoke)

добавить логи/метрики, которые агент умеет читать

ввести exec-plans для задач >2–4 часов


После этого уже можно мерить:

PR/day

время от задачи до PR

сколько багов ловится агентом до человека

сколько раз агент застрял из-за отсутствия контекста/инструмента

Итог (коротко)

Harness Engineering — это не “пишем код ИИ”, а “строим инженерную операционную систему для ИИ-разработки”.

Если хотите реально повторить результат, инвестируйте не в “магический промпт”, а в:

структуру репо

agent-readable docs

архитектурные ограничения

observability

автоматические проверки

циклы обратной связи

регулярную очистку энтропии


Тогда агент начинает не просто генерировать файлы, а стабильно производить инженерный результат.
